# Rule Engine 벤더 조사 — 종합 비교, 초기 데이터 적재 방식 및 추천

**목적:** 향후 PGM 소스 생성(PGM source generation) 트랙을 위한 사전 조사입니다(월요일 확인: *PGM 소스 = 소스코드, 각 PGM은 화면 / 배치작업 / 인터페이스(IF)에 매핑*). 핵심은 각 rule engine이 **초기 데이터를 어떻게 적재하는지**이며, 이와 함께 종합 비교(라이선스, 배포, 가격, 장단점)와 최종 추천을 담았습니다.

> **"초기 데이터" = (A) 룰 정의 + (B) 참조/룩업 테이블.** 런타임 거래 데이터(C)는 요청마다 전달되므로 범위 밖입니다. 벤더 간 차이는 주로 A와 B를 어떻게 작성·적재하는가에 있습니다.

## 1. 종합 비교

| 벤더 | 오픈소스 여부 | 배포 형태 | 가격 모델 | 초기 데이터 적재 방식 | 자동화 적합도* |
|---|---|---|---|---|---|
| **Drools (KIE)** | ✅ Apache 2.0 | 셀프호스트 / 임베디드(Java) | 무료, 유료 지원은 Red Hat 구독 | DRL 파일, 엑셀 디시전 테이블, KieFileSystem/classpath, 시작 시 fact 삽입 | 높음(Java) |
| **IBM ODM** | ❌ 상용 | 셀프호스트, OpenShift/K8s, z/OS, SaaS | 엔터프라이즈 라이선스, 견적형, **매우 높음**(프로세서/VPC) | Decision Center GUI, 룰 프로젝트 import → RuleApp 배포 | 낮음~중간 |
| **Camunda 8 (DMN)** | ⚠️ 부분 — 커뮤니티는 Apache 2.0이나 **8.6+ 운영은 상용 라이선스 필요** | 셀프매니지드 / SaaS | 엔터프라이즈 **연 $50k~$200k+**, SaaS 약 연 $65k(50만 건) | DMN 파일을 classpath 또는 **REST 업로드**로 배포 | 높음(REST) |
| **Progress Corticon** | ❌ 상용 | 셀프호스트 / 임베디드(.js) | 견적형(상용) | 엑셀 import → 룰시트 → `.eds` 컴파일 → 서버 배포 | 중간 |
| **GoRules / Zen** | ✅ MIT(엔진) | **임베디드 / 셀프호스트 / 클라우드** | 엔진 무료, 클라우드/BRMS는 견적형 | **JSON(JDM)** 파일, JDM 에디터, `.json` import/export | **매우 높음(JSON, 다중언어)** |
| **DecisionRules.io** | ❌ 상용 | **SaaS / 프라이빗 클라우드 / 셀프호스트(Docker·K8s, 폐쇄망)** | 무료 티어 → 구독, 셀프호스트는 견적형 | **REST API push**, 1급 룩업 테이블 | **매우 높음(API 우선)** |

*\*자동화 적합도 = 외부 시스템이 룰을 프로그램적으로 생성·적재하기 얼마나 쉬운가(우리 PGM 파이프라인과 직접 연관).*

## 2. 벤더별 상세 (초기 데이터 적재 + 장단점)

### 2.1 Drools (KIE) — *오픈소스 Java 표준*
- **초기 데이터:** 룰은 `.drl` 또는 **엑셀/CSV 디시전 테이블**, `KieFileSystem`/classpath로 적재. 참조 데이터는 시작 시 working memory에 fact로 삽입하거나 DB에서 읽음.
- **✅ 장점:** 무료(Apache 2.0), 성숙·검증됨, 업무 담당자가 편집 가능한 엑셀 디시전 테이블, 대규모 커뮤니티, 모든 JVM 앱에 임베드 가능.
- **❌ 단점:** Java 중심, 학습곡선이 가파름(DRL 문법·RETE), UI 도구(Business Central)가 다소 무겁고 투박, 참조 데이터 초기화는 직접 구현 필요.

### 2.2 IBM ODM — *엔터프라이즈 헤비급*
- **초기 데이터:** Decision Center / Rule Designer에서 작성, 룰 프로젝트 아카이브로 import 후 RuleApp으로 배포, 참조 데이터는 BOM(Business Object Model)을 통해 처리.
- **✅ 장점:** 최고 수준의 거버넌스·버전관리·감사·업무자 작성, 초대형 엔터프라이즈에서 검증, 광범위한 배포 대상(z/OS 포함).
- **❌ 단점:** **매우 고가**(견적형, 프로세서 라이선스), 운영 부담 큼, **프로그램적 초기 적재에 가장 부적합**, 벤더 종속.

### 2.3 Camunda 8 (DMN) — *표준 기반, 프로세스+의사결정*
- **초기 데이터:** DMN `.dmn` 디시전 테이블, classpath 또는 **REST 엔드포인트**("원클릭 배포")로 배포, REST/Java API로 평가.
- **✅ 장점:** OMG **DMN 표준**(이식성), 우수한 REST·동적 배포, BPMN 프로세스 오케스트레이션과 결합, 좋은 모델러.
- **❌ 단점:** 라이선스 강화 — **8.6+ 운영은 상용 라이선스 필요**, 엔터프라이즈 가격이 높음, 순수 rule engine 대비 스택이 무거움.

### 2.4 Progress Corticon — *모델 주도, 로우코드*
- **초기 데이터:** Vocabulary(데이터 모델) + **엑셀 import 룰시트** → Ruleflow → 컴파일된 `.eds` 디시전 서비스로 서버 배포.
- **✅ 장점:** 노코딩, 분석가 친화적, 강력한 룰 검증(충돌·누락 탐지), 자기완결적 `.eds` 산출물, Corticon.js로 브라우저/서버리스 실행 가능.
- **❌ 단점:** 상용(견적형), Drools/Camunda보다 작은 커뮤니티, 모델 주도 방식이 완전 자동 룰 생성에는 유연성이 낮음.

### 2.5 GoRules / Zen — *모던, JSON 네이티브, 다중언어*
- **초기 데이터:** **JSON Decision Model(JDM)** — 의사결정 그래프를 이식 가능한 JSON으로 저장, 오픈소스 JDM 에디터로 작성, Zen 엔진이 직접 로드, 깔끔한 `.json` import/export.
- **✅ 장점:** **엔진이 MIT 오픈소스·셀프호스트 무료(무제한 평가)**, JSON 포맷이 프로그램적 생성에 이상적, **다중언어**(Rust/Node/Python/Go/Java/C#/Swift), 임베드·클라우드 모두 가능, Git 친화적.
- **❌ 단점:** IBM/Drools 대비 생태계가 젊고 작음, 클라우드/BRMS 티어 가격 비공개, 기본 제공 엔터프라이즈 거버넌스 도구가 적음.

### 2.6 DecisionRules.io — *API 우선 SaaS / 셀프호스트*
- **초기 데이터:** 디시전 테이블/트리를 **REST API push**로 관리, 참조 데이터용 1급 **룩업 테이블**(캐싱 지원).
- **✅ 장점:** **API 우선**(자동 룰 생성 파이프라인 연동이 가장 쉬움), 유연한 배포(SaaS / 프라이빗 클라우드 / 폐쇄망 포함 셀프호스트), 무료 티어로 시작, 비개발자 UI.
- **❌ 단점:** 상용(구독 / 셀프호스트는 견적), 신생 벤더, 통신사 데이터 거주성 측면에서 SaaS 고려 필요, 기존 강자 대비 생태계 성숙도 낮음.

## 3. 우리 사용 사례 관점의 판단 기준

우리 시스템은 이미 구조화된 업무 지식을 그래프로 추출합니다. 향후 PGM 트랙은 이를 룰로 변환하고 **자동으로 적재**해야 합니다. 따라서 결정 요인은:
1. **프로그램적 import**(GUI 수작업 없이 생성된 룰을 push 가능한가) → GoRules(JSON), DecisionRules(REST), Camunda(REST) 유리.
2. 연구/PoC 단계의 **오픈소스·비용 통제** → Drools, GoRules 유리.
3. **배포 적합성**(AWS 가능성, 통신사 폐쇄망 가능성) → 셀프호스트 가능 옵션 유리: Drools, GoRules, DecisionRules 셀프호스트, Camunda 셀프매니지드.
4. 운영 전환 시 **한글 처리·엔터프라이즈 거버넌스** → IBM ODM이 최강이나 고가.

## 4. 추천

**1순위 추천: PoC 단계에서는 GoRules(Zen / JDM).**
- 오픈소스(MIT), 셀프호스트 무료, 다중언어이며, **JSON(JDM) 포맷이 추출 지식으로부터 룰을 자동 생성하는 시스템에 가장 적합**합니다. "지식 → 룰 → 적재" 파이프라인을 가장 낮은 비용·마찰로 검증할 수 있습니다.

**유력 대안(API 우선): DecisionRules.io.**
- 비개발자 룰 편집과 **REST 기반 적재**가 가능한 매니지드/로우코드 BRMS를 선호한다면 가장 깔끔한 API 우선 옵션이며, 통신사 데이터 거주성을 위한 셀프호스트도 지원합니다.

**안전한 엔터프라이즈급 대안: Drools.**
- 검증된 완전 오픈소스, Java 네이티브, 엑셀 편집형 디시전 테이블, 벤더 종속 없음을 원한다면 보수적 선택입니다(다만 자체 엔지니어링 부담은 큼).

**고객이 명시적으로 요구할 때만 고려: IBM ODM / Camunda.**
- **IBM ODM** — 엔터프라이즈 거버넌스·감사와 기존 IBM 환경이 고비용·낮은 자동화 적합도를 상회할 때만.
- **Camunda** — DMN 의사결정과 함께 BPMN 프로세스 오케스트레이션이 필요할 때, 그렇지 않으면 강화된 라이선스로 순수 rule engine 용도로는 매력이 낮음.

> **제안 다음 단계:** **GoRules(JSON) + API 우선 옵션 1개(DecisionRules)**로 적재 경로를 프로토타이핑 — 추출 지식에서 소규모 룰셋을 생성·적재·평가합니다. 거의 제로에 가까운 라이선스 비용으로, 전용 PGM 설계 단계 이전에 실현 가능성을 검증합니다.
